# J_0007 Real card interaction journey

<!-- METADATA:TYPE=J,ID=J_0007,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=4 -->

## Goal

Validate the real user journey for Feishu question cards in an intern group:

- the user asks an intern, in natural language, to send a card;
- the card is visible as a single pending question;
- the user clicks a button or submits a form;
- the intern receives the answer and replies naturally;
- the pending question is cleared after each answer.

The journey covers a simple button choice, free text input, and a mixed
single-select plus multi-select form.

## Why J

This is J-scoped because success depends on a real agent turn and a real
user-visible card journey. F cases can validate callback routes, card schema, or
daemon pending-state contracts, but they cannot prove that an intern responds to
the user's selected answer after a natural prompt.

## Cost profile

Paid/agent. The case starts a real Codex intern session, sends natural prompts,
waits for Codex replies, and validates the answers from Feishu-visible and
daemon/report evidence.

## Required environment

- A lead-prepared local or throwaway workspace suitable for card interaction
  tests.
- Backend: `codex`.
- The workspace mode must be fixed at add time; this case must not add, delete,
  migrate, or switch workspace mode.
- The CI runner may create only case-scoped intern, session, group, task/report,
  and temporary artifacts.

## Case resources

- namespace: `ci_j_0007`
- intern: `intern_ci_j_0007_codex_<run_id>`
- task fixture: `task_ci_j_0007_card_<run_id>`
- report prefix: `j_0007_card_interaction`
- retained scene: the final Feishu group, visible cards, intern replies, daemon
  pending-answer evidence, and CI report artifacts remain available for review.

Start-of-case cleanup may remove only the `ci_j_0007` namespace from previous
runs. It must not clean lead-owned workspaces, shared debug deployments, or
other cases' resources.

## Implementation prerequisites

Do not weaken this journey into low-level callback API checks. If helpers are
missing, implementation must wait for or add CI helpers for:

- finding one pending question card after a user prompt;
- clicking a Feishu card button as a user-visible action;
- submitting a Feishu card form with free text, single-select, and multi-select
  fields;
- waiting for callback application and pending-question cleanup;
- correlating the intern reply with the card answer.

## Key assertions

- Each user prompt creates exactly one pending question card.
- Card controls match the user's request: A/B buttons, free input, single-select,
  and multi-select.
- Each card action has visible callback evidence and daemon pending-answer
  evidence.
- The intern reply reflects the selected or submitted answer and does not mix in
  unselected options.
- Pending questions are cleared after each answer.
- The case never adds, deletes, migrates, or switches the workspace.

## Failure classification

- Multiple pending cards for one prompt: product bug.
- Missing requested controls: product bug or missing product capability.
- Card click or form submit cannot be represented as a real user-visible action:
  CI helper gap.
- Callback succeeds but the intern never receives or uses the answer: product
  bug.
- Pending question remains after answer: product bug.
- Workspace cleanup, deletion, migration, or mode switching occurs: CI bug.
