## Retired reason

Session 12 consolidation folded this multi-attachment order path into
`J_0021`, the inbound attachment bundle. Ordered tokens and blocker
identification are now covered as Phase 4 of `J_0021`.

## Concise proposal

Do: send two attachments in order: a text file and an image/file with a second
token.

Prompt: ask the intern to process both attachments in the same order.

Look for: both attachments are persisted, injected together, processed in
message order, and consumed once.

## Goal

Validate multi-attachment user behavior:

- multiple attachments from one or adjacent Feishu messages are preserved;
- pending attachment order is stable;
- a single prompt can consume the whole bundle;
- the intern's answer names each attachment and token in order.

## Why J

Attachment array parsing is F. This journey proves the user can send a small
bundle and ask the intern to process it coherently.

## Key assertions

- Attachment order is preserved.
- The bundle is consumed once.
- The intern does not skip or merge distinct attachments.

## Implementation prerequisites

- Multi-attachment driver.
- Pending attachment order reader.
- Reply order classifier.

## Failure classification

- Attachments reorder or one is dropped: product bug.
- Bundle is consumed twice: product bug.
- Agent ignores one attachment: product/model behavior finding.
