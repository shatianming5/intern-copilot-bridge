# J_0026 Busy receiver queued peer message journey

<!-- METADATA:TYPE=J,ID=J_0026,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=8 -->

## Concise proposal

Prompt: start a receiver intern on a slow task.

Prompt: ask a sender intern to send the receiver a non-interrupting next-turn
message.

Look for: delivery is queued while receiver is Working, then appears to the
receiver after the current turn finishes.

## Goal

Validate the user-visible busy receiver journey:

- receiver is actively Working;
- sender uses a natural prompt to send a peer message in next-turn mode;
- daemon records queued delivery instead of forcing an unsafe live interrupt;
- receiver sees the message after the active turn completes;
- receiver replies to the queued message in the expected scope.

## Why J

`J_0014` covers peer-send routing and basic errors. This case focuses on the
real user story for busy receivers: "send it for their next turn, do not break
what they are doing."

## Cost profile

Paid/agent. The case starts two real Codex sessions and a slow receiver turn.

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Clean only `ci_j_0026` resources. | Previous sender/receiver groups, sessions, queued messages, and reports are removed or isolated. |
| 2 | action | Create sender and receiver Codex interns and groups. | Both sessions are live and project-scoped. |
| 3 | action | Prompt receiver to run a slow harmless check. | Receiver enters Working. |
| 4 | action | Prompt sender: "请给 receiver 发一条下轮处理的消息：queued token。" | Sender uses natural language, not direct daemon API. |
| 5 | wait/assert | Wait for sender completion. | Sender reports the message was queued/delivered for next turn; daemon evidence says queued while receiver was busy. |
| 6 | wait/assert | Wait for receiver slow turn to finish. | Original receiver task completes without interruption. |
| 7 | wait/assert | Wait for queued message visibility. | Receiver pane/group sees the queued token after the active turn. |
| 8 | action | Prompt receiver: "请处理刚才收到的 queued token。" | Follow-up is delivered in receiver group. |
| 9 | wait/assert | Wait for receiver reply. | Receiver replies with the queued token and correct sender/project evidence. |
| 10 | assert | Check report. | Report links sender prompt, daemon queue evidence, receiver turn timing, queued visibility, and final reply. |

## Key assertions

- Busy receiver is not interrupted by the peer message.
- Peer message is queued for the next turn and becomes visible.
- Scope is `(project, receiver)`; no same-name leakage.

## Implementation prerequisites

- Multi-intern prompt/reply helper.
- Active-turn and queued-peer evidence.
- Receiver pane/group visibility extractor.

## Failure classification

- Peer message interrupts an active receiver turn: product bug.
- Queued message is lost after receiver finishes: product bug.
- Sender falsely reports success without delivery evidence: product bug.
- CI cannot create a stable busy window: CI helper gap.
