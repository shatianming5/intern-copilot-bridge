# J_0019 Pending restart live conversation journey

<!-- METADATA:TYPE=J,ID=J_0019,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=7 -->

## Concise proposal

Prompt: start a real Codex turn that runs a slow harmless check.

Do: trigger a case-scoped policy/env update that requires this Codex session to
restart while the turn is still active.

Look for: the product records a pending restart instead of killing the working
turn.

Prompt: after the first turn finishes and the pending restart completes, ask a
short follow-up in the same Feishu group.

Look for: the restarted session answers, the group/chat binding is unchanged,
and restart evidence points to the policy marker.

## Goal

Validate the user-visible pending restart journey:

- a real Codex turn is active;
- a case-scoped policy/env update requires a restart;
- the active turn is not interrupted;
- the restart is queued as pending while the intern is Working;
- after the turn finishes, the pending restart completes;
- the user can continue in the same Feishu group after restart.

## Why J

`F_0034` validates policy sync, env materialization, Idle Codex auto-restart,
and replay de-duplication without creating a real Working conversation. This
case is J-scoped because it proves the user journey where a supervisor is
actively talking to an intern while the restart requirement arrives.

## Cost profile

Paid/agent. The case starts a real Codex turn, waits through a policy-triggered
pending restart, and sends a follow-up prompt after restart.

## Required environment

- Backend: `codex`.
- A case-scoped Feishu group and Codex session.
- A safe policy/env marker driver that can require restart without using
  configuration webviews, packaging, deploy, bootstrap, or relay restart.
- Restart progress evidence from daemon/relay websocket, daemon state, or an
  accepted equivalent source.

## Key assertions

- Working turn is not killed by a restart-required policy update.
- Pending restart status is visible while the intern is Working.
- Restart happens after the original turn finishes.
- Follow-up prompt works in the same group after restart.
- Restart evidence is tied to the case marker and redacts secrets.

## Implementation prerequisites

- Active-turn waiter with enough timing evidence to trigger policy update while
  the turn is still running.
- Safe case-scoped policy/env marker driver.
- Restart progress collector for pending and completed states.
- Follow-up prompt/reply helper that can prove same group and same intern scope.

## Failure classification

- Working turn is interrupted or killed by restart: product bug.
- Restart-required policy update is ignored: product bug or policy driver gap,
  classified by marker evidence.
- Pending restart is never processed after the turn finishes: product bug.
- Follow-up prompt goes to the wrong group/session or cannot resume: product
  bug.
- Restart evidence leaks secret env material: CI/product security bug.
- Runner cannot safely trigger a case-scoped restart marker: CI helper gap.
