# J_0041 Main-bot status to group follow-up journey

<!-- METADATA:TYPE=J,ID=J_0041,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=8 -->

## Concise proposal

Do: ask the main bot for readonly status/list information about a case intern.

Prompt: use that visible status to send a follow-up in the intern group.

Look for: main-bot status is consistent with the group that actually answers,
and main-bot readonly commands do not themselves start agent work.

## Goal

Validate a user journey spanning main-bot readonly status and intern group use:

- main bot reports status/list/debug information for the target intern;
- readonly main-bot interaction does not start an agent turn;
- user then prompts the intern group;
- intern group answer matches the status main bot advertised.

## Why J

Main-bot readonly command contracts are F. This case adds the real user flow:
check status in main bot, then continue in the intern group.

- namespace: `ci_j_0041`

- follow-up token: `CI_J0041_GROUP_<run_id>`

## Key assertions

- Main-bot readonly status is accurate.
- Readonly status does not trigger agent work.
- User can act on status to reach the correct group.

## Implementation prerequisites

- Main-bot readonly driver.
- No-turn evidence for main-bot request.
- Group consistency checker.

## Failure classification

- Main bot reports stale/wrong group: product bug.
- Readonly command starts agent turn: product bug.
