# J_0018 Running supplement latest-instruction journey

<!-- METADATA:TYPE=J,ID=J_0018,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=7 -->

## Concise proposal

Prompt: start a real Codex turn that does a slow, harmless check and says it
will answer in bullets.

Do: while the turn is still running, send a new user message that changes the
requested output to a table and adds a new token.

Look for: the final answer honors the newer running supplement, does not cling
to the old bullet-only request, and the report links the supplement message to
the final turn evidence.

## Goal

Validate the real running-supplement journey:

- a user starts a long-running Codex turn;
- the user sends an additional instruction while that turn is active;
- the active turn receives the supplement through the product mechanism;
- the final answer follows the newest instruction;
- report evidence proves the final answer is not an older ghost response.

## Why J

Running supplement behavior is inherently about live user/agent interaction
while a turn is already active. F can validate hook parsing or context payload
shape, but only J can prove that a real agent turn receives and follows the
latest user instruction.

## Cost profile

Paid/agent. The case starts a real Codex turn, injects a second user message
while it is active, and waits for the final answer.

## Required environment

- Backend: `codex`.
- A case-scoped Feishu group and Codex session.
- A harmless slow probe or prompt pattern that reliably leaves enough time to
  send a supplement while the turn is running.
- Turn transcript evidence that can link the supplement to the active turn.

## Key assertions

- Supplement is sent while the turn is active, not before or after it.
- Final answer follows the newest instruction.
- The old prompt is not blindly replayed as the only instruction.
- Report evidence ties supplement message id to the active turn.
- No stale final answer overwrites or follows the correct supplement-aware
  answer.

## Implementation prerequisites

- Active-turn waiter with timing evidence.
- Running-supplement sender that uses the real product path.
- Transcript/timeline extractor that can prove supplement visibility.
- Format classifier for bullet-only vs table-like final answer.

## Failure classification

- Supplement is delivered but agent ignores it: product bug or model behavior
  finding, classified by transcript evidence.
- Supplement is not attached to the active turn: product bug.
- Final timeline shows stale answer after supplement-aware answer: product bug.
- The runner cannot reliably send while turn is active: CI helper gap.
