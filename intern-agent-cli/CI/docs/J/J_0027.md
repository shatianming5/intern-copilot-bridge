# J_0027 Same-name cross-project peer scope journey

<!-- METADATA:TYPE=J,ID=J_0027,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=8 -->

## Concise proposal

Do: create two workspaces/projects that both contain an intern with the same
name.

Prompt: ask a sender to message the receiver in its own project.

Look for: only the receiver in the intended project sees the message; ambiguous
project-less sends are rejected or clarified.

## Goal

Validate project-scoped peer delivery in a real journey:

- same intern name exists in two projects;
- natural peer-send from a sender in project A targets project A receiver;
- project B same-name intern does not receive it;
- project-less ambiguous requests do not silently pick one target;
- final report proves `(project, internName)` scope.

## Why J

F can validate registry compound keys. This case proves user-visible messaging
does not cross projects when agents use the peer-send skill.

## Cost profile

Paid/agent. The case uses multiple live sessions, but prompts are small.

## Key assertions

- Peer delivery respects `(project, internName)`.
- Same-name intern in another project is unaffected.
- Ambiguity is visible and safe.

## Implementation prerequisites

- Multi-project live intern fixture.
- Peer-send skill evidence.
- Receiver timeline isolation helper.

## Failure classification

- Message crosses to wrong project: product bug.
- Project-less ambiguous send silently picks a target: product bug.
- Sender cannot express scoped peer-send naturally: CI/helper gap or product UX gap.
