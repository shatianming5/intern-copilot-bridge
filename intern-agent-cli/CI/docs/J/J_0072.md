# J_0072 Peer message with attachment unsupported honesty journey

## Scope note

Keep this case only as builtin `peer-send`/collaboration contract coverage. It
may assert that a structured unsupported tool result is surfaced without fake
delivery, but it must not grade generic alternative-suggestion style.

## Concise proposal

Prompt: ask an intern to send a file attachment to another intern via peer-send.

Look for: if peer attachments are unsupported, the sender says so honestly and
offers a supported alternative; it must not claim delivery falsely.

## Goal

Validate unsupported peer-attachment handling:

- user asks for peer attachment send;
- product/skill path does not support attachments or returns explicit reason;
- intern reports the limitation and suggests Feishu file/group path;
- receiver does not falsely show delivered attachment.

## Why J

The daemon reason is F-level. This journey validates the live builtin-skill path
when a user asks for an unsupported collaboration action.
