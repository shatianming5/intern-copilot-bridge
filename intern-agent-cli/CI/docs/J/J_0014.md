# J_0014 Peer send routing and error contract

<!-- METADATA:TYPE=J,ID=J_0014,STATUS=Implemented,OWNER=intern_ci_lead,SESSION=112 -->

## Goal

Validate peer send routing when live sender and receiver intern sessions exist:
same-project delivery, receiver-pane visibility, missing target, ambiguous
target, and invalid mode errors.

## Why J

This case starts live intern sessions and checks the receiver tmux pane for the
peer message token. That is no longer treated as F-level API-only evidence.

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Seed sender and receiver sessions in project A. | Both sessions exist and have chat mappings. |
| 2 | action/assert | Send `mode=next` peer message from sender to receiver. | API returns `delivered` or `queued`, and receiver pane contains the token. |
| 3 | action/assert | Send to a missing target in project A. | API returns `undeliverable` with `unknown_target`. |
| 4 | action/assert | Create same receiver name in project B, then send without explicit project. | API returns `undeliverable` with `ambiguous_target` and candidates. |
| 5 | action/assert | Send with invalid mode. | Daemon returns HTTP 400 `invalid_mode`. |
