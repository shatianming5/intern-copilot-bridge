# J_0016 Review feedback update-before-merge journey

<!-- METADATA:TYPE=J,ID=J_0016,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=7 -->

## Concise proposal

Prompt: assign a tiny protected-repo task and ask the intern to continue.

Look for: the intern opens one MR/PR and does not merge yet.

Prompt: "review 要求改成另一句，请更新同一个 MR/PR，不要 merge。"

Look for: the intern pushes a new revision to the same change request; target
branch still has not moved.

Prompt: "现在可以 merge。"

Look for: merge happens only after approval, includes the revised content, and
closeout is clean.

## Goal

Validate the real review-feedback journey for protected work:

- user assignment starts the task;
- the intern opens a protected MR/PR;
- the supervisor gives review feedback before merge;
- the intern updates the same MR/PR rather than opening a duplicate or merging;
- the supervisor explicitly approves merge;
- target branch advances only after approval and contains the revised content.

## Why J

`J_0001` and `J_0002` cover the straight assignment -> continue -> approve
merge lifecycle. This case is J-scoped because it inserts the real review
feedback loop that supervisors use daily: request changes, verify the same
change request updates, then approve merge.

## Cost profile

Paid/agent. The case starts a real Codex intern session, sends natural review
feedback, creates and updates a real provider change request, and may merge into
a throwaway protected test target.

## Required environment

- Backend: `codex`.
- A lead-prepared protected Codeup or GitHub test workspace.
- One run should target a single provider; provider-specific variants can be
  split later if review wants separate numbers.
- The target branch must be approved for CI merge.
- The case must not add, delete, migrate, or switch workspace mode.

## Task fixture

The task body should require one small business-only edit:

```text
Please update README.md under "CI review journey" with:
initial review journey line.

Do not modify .intern_workspace.
```

The review prompt later changes the expected line to:

```text
revised review journey line.
```

## Key assertions

- Review feedback updates the same change request.
- Target branch does not move before explicit merge approval.
- The final merged content reflects the review feedback, not the initial draft.
- Business diff excludes `.intern_workspace`, status, history, and task
  knowledge.
- Merge happens through the provider change-request path.

## Implementation prerequisites

- Provider change-request revision evidence for Codeup/GitHub.
- A way to distinguish same MR/PR update from duplicate MR/PR creation.
- Focused diff evidence before and after review feedback.

## Failure classification

- Intern merges before review approval: product bug.
- Intern opens duplicate MR/PR instead of updating the existing one: product
  bug.
- Target branch advances before approval: product bug.
- Review feedback is ignored or final content is stale: product bug.
- Metadata enters the business change request: product bug.
- Provider revision evidence is unavailable: CI helper gap.
