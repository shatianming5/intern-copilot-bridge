# J_0020 Machine helper live diagnosis journey

<!-- METADATA:TYPE=J,ID=J_0020,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=7 -->

## Concise proposal

Do: from a normal intern group, start a machine helper for an available debug
machine.

Prompt: in the helper group, ask for a harmless read-only diagnosis summary.

Look for: the helper replies with the machine/context it is helping with and
does not expose secrets.

Do: invite the owner, stop the helper, then start it again.

Prompt: send a short follow-up to the restarted helper group.

Look for: owner visibility is correct, stop removes the old active runtime, and
restart creates or rebinds a usable helper group rather than pointing at a stale
chat id.

## Goal

Validate the real user journey for machine helper operation:

- supervisor starts a helper from the normal Feishu helper surface;
- a helper runtime and helper Feishu group become usable;
- the helper receives machine/operator/issue context;
- the user can ask a harmless diagnosis question and receive a useful answer;
- owner invitation is visible and scoped to the helper group;
- stop and restart do not leave a stale group/chat binding.

## Why J

F cases can validate `/helper start|stop|invite-owner` card contents, callback
payloads, daemon start/stop APIs, registry mutation, and stale `chat_id`
handling. This case is J-scoped because it proves that the started helper is a
real agent the user can talk to, and that the stop/restart lifecycle preserves a
usable user conversation surface.

## Cost profile

Paid/agent. The case starts a machine helper runtime, sends real helper-group
prompts, waits for helper replies, and verifies Feishu group visibility.

## Required environment

- Existing deployed debug machine with helper support.
- A case-scoped normal intern group that can send `/helper start`, `/helper
  invite-owner`, and `/helper stop`.
- A helper runtime backend supported by the deployed product, usually Codex.
- No package, deploy, bootstrap, relay restart, configuration webview, or shared debug
  cleanup.

## Key assertions

- Helper start creates a real usable helper runtime and group.
- Helper replies to a harmless user prompt with expected context and without
  leaking secrets.
- Invite-owner is scoped to the helper group.
- Stop makes the old active helper binding unusable.
- Restart creates/rebinds a valid helper group and does not point to a stale
  deleted chat id.

## Implementation prerequisites

- Real helper start/stop/invite-owner driver, or an accepted source-driver that
  reaches the same relay-local helper branches.
- Helper group timeline reader.
- Runtime/session registry evidence for machine helper sessions.
- Secret redaction classifier for helper replies.

## Failure classification

- Helper start card/API reports success but no usable helper group exists:
  product bug.
- Helper cannot answer a normal group prompt after start: product bug.
- Helper reply leaks secret material: product/security bug.
- Invite-owner affects the wrong helper group or unrelated users: product bug.
- Stop leaves a stale active helper binding: product bug.
- Restart reuses a deleted/stale chat id or cannot receive follow-up prompt:
  product bug.
- CI cannot drive real helper card actions or read helper group timeline:
  CI helper gap.
