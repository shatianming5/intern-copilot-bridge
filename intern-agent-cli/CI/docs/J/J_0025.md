# J_0025 Request-user-input duplicate suppression journey

<!-- METADATA:TYPE=J,ID=J_0025,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=8 -->

## Concise proposal

Prompt: ask Codex to call `request_user_input` for one concrete decision.

Look for: Feishu shows one effective question card, even if Codex/daemon sees
multiple internal request-user-input signals.

Do: answer the card.

Look for: Codex receives the answer once and continues without duplicate stale
cards.

## Goal

Validate real duplicate suppression for Codex request-user-input:

- Codex creates one user decision;
- Feishu receives one active card for that decision;
- duplicate PreTool/TUI watcher events are adopted or suppressed;
- answering the card closes the pending question;
- the intern receives the answer exactly once.

## Why J

The duplicate-card bug only matters when a real user sees multiple cards and
clicks one. Source-level de-duplication alone is not enough.

## Cost profile

Paid/agent. The case drives real Codex `request_user_input` and Feishu card
answer flow.

- namespace: `ci_j_0025`

- decision token: `CI_J0025_DECISION_<run_id>`

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Clean only the `ci_j_0025` namespace. | Previous intern, group, pending questions, and reports are removed or isolated. |
| 2 | action | Create Codex intern, session, and Feishu group. | Group is green and no pending question exists. |
| 3 | action | Prompt: "请用一个选择卡问我要不要输出 A 或 B；等我选择后只输出所选项和 decision token。" | Prompt induces one `request_user_input` decision. |
| 4 | wait/assert | Wait for decision card. | Exactly one active card is visible for this decision; internal duplicate events are either absent or linked as adopted. |
| 5 | action | Click/select `A`. | Card answer is delivered once. |
| 6 | wait/assert | Wait for Codex reply. | Reply contains `A` and the decision token, not `B`. |
| 7 | assert | Check pending/card state. | No duplicate active card remains and no second answer is injected. |
| 8 | assert | Check report. | Report links transcript/tool events, card ids, adoption/suppression evidence, answer id, and final reply. |

## Key assertions

- One logical decision produces one active user-facing card.
- Duplicate internal request paths do not create duplicate active cards.
- Answer is delivered exactly once.
- Pending state is project/intern scoped.

## Implementation prerequisites

- Real Codex request-user-input prompt driver.
- Card timeline and pending question state reader.
- Duplicate/adoption event extractor.

## Failure classification

- Two active cards appear for one decision: product bug.
- Answer goes to the wrong pending key: product bug.
- Codex receives duplicate answers: product bug.
- CI cannot reliably induce request-user-input: CI helper gap.
