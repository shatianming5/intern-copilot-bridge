# J_0021 Feishu inbound attachment bundle journey

<!-- METADATA:TYPE=J,ID=J_0021,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=12 -->

## Concise proposal

Do: send file, image, attachment-only, and multi/rich attachment inputs in
separate phases with clear wait boundaries.

Prompt: ask concrete business questions for each phase: release risks/owner,
image token/number, CSV high-priority count, and ordered blocker tokens.

Look for: each attachment is consumed by the right prompt exactly once, no phase
leaks into the next one, image unsupported behavior is honest, and multi-content
order is stable.

## Goal

Validate inbound Feishu attachment behavior as one user journey bundle:

- a markdown attachment with same-message instructions is usable immediately;
- an image attachment with same-message instructions is read or honestly
  reported unsupported;
- an attachment-only message does not start an empty ambiguous turn and can be
  consumed by the next explicit prompt;
- multiple attachments or rich-post content preserve order;
- old attachments are not re-consumed across phases.

## Why J

F can validate extraction, download, persistence, and prompt-hook source
contracts. This case is J-scoped because a real user sends several attachment
forms and expects useful, honest answers without timing confusion.

## Cost profile

Paid/agent. The case sends a real Feishu attachment, prompts Codex, and waits
for a real reply.

## Key assertions

- Each phase has a clear message id and wait boundary before the next phase.
- Same-message text+attachment inputs are consumed in that turn.
- Attachment-only input does not start an empty turn and is consumed by the next
  explicit prompt.
- Image unsupported behavior is honest; wrong token guesses fail.
- Multi-attachment/rich content order is preserved.
- Later turns do not silently reuse stale attachments.
- File paths and checksums are recorded without leaking secrets.

## Implementation prerequisites

- Real Feishu file/image/rich-post attachment driver or accepted source-driver.
- Inbox/pending attachment state reader.
- Prompt/reply helper with token evidence.
- No-turn waiter for attachment-only messages.
- Reply classifiers for risk/owner extraction, image token/unsupported outcome,
  CSV count, and ordered token/blocker output.

## Failure classification

- Attachment download or persistence fails: product bug or Feishu permission gap.
- Same-message attachment+text is not exposed to the prompt: product bug.
- Attachment-only starts an empty turn: product bug.
- Intern guesses image content despite unsupported vision: product/model behavior
  finding.
- Attachments reorder, one is dropped, or stale attachments are re-consumed:
  product bug.
- CI sends phases without waiting for stable boundaries: CI script bug.
