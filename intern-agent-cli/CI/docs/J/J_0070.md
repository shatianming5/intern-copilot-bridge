# J_0070 Coordinator status rollup with busy worker journey

<!-- METADATA:TYPE=J,ID=J_0070,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=9 -->

## Concise proposal

Prompt: while one worker is busy and one is idle, ask coordinator for team
status.

Look for: coordinator distinguishes busy/idle states and does not claim busy
worker finished.

## Goal

Validate team status rollup:

- worker A is Working;
- worker B is Idle or complete;
- user asks coordinator for status;
- coordinator reports accurate per-worker state and next wait/action.

## Why J

This is the user-visible status view of a live team, beyond static metadata.

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Create team and start worker A on slow task. | Worker A Working; worker B idle/complete. |
| 2 | action | Prompt coordinator: "汇总当前 team 状态。" | Status request delivered. |
| 3 | wait/assert | Wait for rollup. | Coordinator reports A busy and B idle/complete accurately. |
| 4 | assert | Check evidence. | Report links worker state snapshots and coordinator reply. |

## Key assertions

- Busy worker is not reported as done.
- Status is per-worker, not flattened.

## Implementation prerequisites

- Worker state snapshot helper.
- Coordinator prompt/reply evidence.

## Failure classification

- Coordinator fabricates completion: product/model behavior bug.
