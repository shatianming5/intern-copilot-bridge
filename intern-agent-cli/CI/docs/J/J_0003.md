# J_0003 GitHub repo_dotdir to metadata_branch migration journey

<!-- METADATA:TYPE=J,ID=J_0003,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=6 -->

## Concise proposal

Do: start from an exclusive GitHub workspace in `repo_dotdir` mode with real
task/status/history/knowledge/skill metadata.

Do: run `internctl workspace migrate-mode --repo-url <repo> --target metadata_branch`.

Look for: a GitHub migration PR opens against the metadata branch and contains
the real `.intern_workspace` tree, not only a marker.

Do: merge the migration PR, then re-add or reuse the workspace in
`metadata_branch` mode.

Look for: task/status/history/knowledge/skill metadata resolves from the
metadata branch, Feishu group routing still targets the same project/intern
scope, and no old in-place mode switch path was used.

## Goal

Validate the replacement user journey for migrating a GitHub workspace from
`repo_dotdir` to `metadata_branch` mode:

- the old relay workspace record is released for the case scope;
- `internctl workspace migrate-mode` creates a real GitHub PR;
- the PR moves the real `.intern_workspace` metadata tree to the metadata
  branch;
- the user/operator merges the PR;
- the workspace is re-added or reused in `metadata_branch` mode;
- metadata and Feishu routing still work after migration.

## Why J

The old J_0003 in-place mode switch story is retired. This replacement is
J-scoped because it is an end-to-end operator journey across workspace records,
GitHub PR review/merge, workspace re-add/reuse, metadata resolution, and
user-visible intern routing. F cases can validate individual CLI guards and
migration fixtures; this J validates the full migration experience.

## Cost profile

Operator journey. It may not require paid agent output, but it does require a
real protected workflow surface: GitHub PR creation, user/operator merge, and
workspace re-add/reuse. It must run only in an exclusive throwaway workspace and
repository prepared for migration tests.

## Required environment

- Provider: `github`.
- Initial workspace mode: `repo_dotdir`.
- Target workspace mode: `metadata_branch`.
- A dedicated non-production GitHub repository with a configured metadata
  branch.
- Exclusive ownership of the test workspace and relay record for the duration
  of the case.
- No `local_only` workspace is involved.

## Case resources

- namespace: `ci_j_0003`
- source workspace: `ci_j_0003_github_repo_dotdir_<run_id>`
- migration PR branch: `ci_j_0003/migrate_metadata_branch_<run_id>`

## Key assertions

- Migration is explicit: release old record, create GitHub PR, merge PR, then
  re-add/reuse workspace.
- The migration PR contains real `.intern_workspace` content.
- Metadata resolves from `metadata_branch` after the workspace is re-added or
  reused.
- Feishu routing uses the new workspace record and remains scoped to the same
  project/intern identity.
- No old in-place workspace mode switch command or relay/daemon mode endpoint is
  used.

## Failure classification

- Migration PR is missing or targets the wrong branch: product bug.
- Migration PR contains only markers or loses metadata files: product bug.
- Re-add/reuse cannot resolve metadata from `metadata_branch`: product bug.
- Feishu routing points at the released old record: product bug.
- Any old in-place mode switch path is used: CI bug.
- Cleanup mutates shared or lead-owned workspaces: CI bug.
