# J_0071 Cross-team permission boundary journey

<!-- METADATA:TYPE=J,ID=J_0071,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=9 -->

## Concise proposal

Prompt: ask a coordinator to assign work to a worker from another team.

Look for: product refuses or asks for proper reassignment; it must not silently
cross team ownership.

## Goal

Validate cross-team permission boundary:

- two teams exist;
- coordinator A targets worker from team B;
- action is refused or requires explicit team reassignment;
- no worker B task/goal is mutated by coordinator A.

## Why J

F can validate role APIs. This journey proves the agent-visible workflow
respects team boundaries.

- namespace: `ci_j_0071`

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Create two isolated teams. | Role/team metadata explicit. |
| 2 | action | Prompt coordinator A to assign token task to worker B. | Request crosses boundary. |
| 3 | wait/assert | Wait for refusal/clarification. | Product refuses or asks for reassignment; worker B unchanged. |
| 4 | assert | Check report. | Report links request, boundary evidence, and no-mutation snapshots. |

## Key assertions

- Cross-team mutation is not silent.
- Worker outside assigned team is unchanged.

## Implementation prerequisites

- Two-team fixture.
- Worker mutation guard.

## Failure classification

- Cross-team task/goal mutation succeeds silently: product bug.
