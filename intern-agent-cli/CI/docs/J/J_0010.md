# J_0010 Team coordinator delegation journey

<!-- METADATA:TYPE=J,ID=J_0010,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=6 -->

## Concise proposal

Do: create a Codex coordinator, a Codex team lead, and one Codex worker from
the Team Mode UI or equivalent CLI-backed commands.

Prompt coordinator: "请把这个小检查任务交给 runtime team，并在 worker 接受后汇总给我。"

Look for: coordinator/team/team lead/worker nodes appear in TreeView with the
right roles, the team lead creates a worker task, peer-send delivers the
assignment, and the worker receives the task in its real session.

Prompt worker: `接受这个 task，先回复你准备怎么做。`

Look for: worker status moves to the task, worker replies naturally in Feishu,
and coordinator/team lead can summarize the delegation without flattening team
metadata or bypassing role boundaries.

## Goal

Validate the real user journey for Team Mode delegation:

- create or select a user-owned coordinator;
- create a team with team lead and worker roles;
- bind the team to the coordinator;
- ask the coordinator to delegate a small task;
- have the team lead create worker task metadata and notify the worker;
- verify the worker accepts through a real prompt and replies in its session;
- keep TreeView, metadata, Feishu groups, and peer-send evidence aligned.

## Why J

F can validate TreeView context values, `internctl team create`, rollback, role
metadata, and `assign-worker-task` in isolation. This case is J-scoped because
it follows the supervisor's actual workflow: the user asks a coordinator to
organize a team, the coordinator/team lead route work to a worker, and the
worker responds in a live intern session.

## Cost profile

Paid/agent. The case starts real Codex sessions for at least coordinator, team
lead, and worker, sends natural prompts, and waits for real agent replies.

## Required environment

- Backend: `codex` for coordinator, team lead, and worker.
- A case-scoped local or throwaway workspace with Team Mode enabled.
- Real Feishu groups for every created intern.
- Peer-send delivery through the local daemon, not direct mailbox writes.
- Team creation rollback helpers must already be stable enough that this J does
  not leave partial lead/worker/groups on fixture creation failure.

## Key assertions

- Team creation is metadata-first and does not leave partial Feishu/session
  side effects if fixture creation fails.
- Coordinator, team lead, and worker roles remain distinct in metadata,
  TreeView, session registry, and Feishu group routing.
- Worker task creation uses the team assignment contract and records the
  correct team/lead/worker relationship.
- Peer-send delivery is real daemon delivery and is visible to the worker.
- Worker acceptance and reply happen in a real worker session after a natural
  prompt.
- Cleanup stays inside the `ci_j_0010` namespace.

## Implementation prerequisites

- A J helper to create and clean case-scoped Team Mode resources safely.
- A TreeView snapshot/assert helper for coordinator/team grouping.
- A peer-send delivery waiter that records delivered vs queued vs failed.
- A live multi-intern prompt/reply helper that can observe coordinator, team
  lead, and worker sessions independently.

## Failure classification

- Team create leaves lead/worker/groups without `team.json`: product bug if
  fixture creation used the product path; CI bug if cleanup/preflight bypassed it.
- Coordinator/team roles flatten or disappear in TreeView: product bug.
- `assign-worker-task` creates malformed task metadata: product bug.
- Peer notification is not delivered or targets the wrong worker/project:
  product bug or daemon routing bug.
- Worker cannot accept or status points at the wrong task: product bug.
- Multi-intern prompt/reply evidence cannot be captured: CI helper gap.
