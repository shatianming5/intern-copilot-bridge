# J_0023 Attachment-only then text follow-up journey

<!-- METADATA:TYPE=J,ID=J_0023,STATUS=Retired,OWNER=intern_ci_lead,SESSION=12 -->

## Retired reason

Session 12 consolidation folded this attachment-only follow-up path into
`J_0021`, the inbound attachment bundle. Attachment-only no-empty-turn and
next-prompt consumption are now covered as Phase 3 of `J_0021`.

## Concise proposal

Do: send an attachment-only message to the intern group.

Prompt: after the product acknowledges receipt, send a text instruction that
refers to "that attachment".

Look for: no turn starts on the attachment-only message, the next text consumes
the pending attachment exactly once, and the intern answers from the file.

## Goal

Validate the attachment-only user path:

- attachment-only Feishu messages are persisted;
- the product asks or waits for the user to send an instruction;
- no empty prompt turn is started;
- the next text prompt receives the pending attachment context;
- pending attachments are consumed exactly once.

## Why J

This combines Feishu message routing, prompt-hook behavior, and a real user
follow-up. It cannot be proven by persistence alone.

## Cost profile

Paid/agent. The case sends a real attachment-only message, then a real prompt.

## Key assertions

- Attachment-only does not start an empty/ambiguous agent turn.
- Next text prompt consumes the pending attachment.
- Attachment is not lost or consumed twice.

## Implementation prerequisites

- Attachment-only Feishu driver.
- No-turn waiter.
- Inbox state and prompt/reply evidence helpers.

## Failure classification

- Attachment-only starts an empty turn: product bug.
- Attachment is lost before follow-up: product bug.
- Attachment is consumed more than once: product bug.
- Absence-of-turn evidence is too weak: CI helper gap.
