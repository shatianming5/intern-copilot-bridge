# J_0022 Feishu image attachment inspection journey

<!-- METADATA:TYPE=J,ID=J_0022,STATUS=Retired,OWNER=intern_ci_lead,SESSION=12 -->

## Retired reason

Session 12 consolidation folded this image attachment path into `J_0021`, the
inbound attachment bundle. Image token reading or honest unsupported behavior
is now covered as Phase 2 of `J_0021`.

## Concise proposal

Do: send a small generated image attachment that visibly contains a token.

Prompt: ask the intern to inspect the attached image and describe the token.

Look for: the image lands in `.feishu_inbox`, the intern uses image-viewing
capability or clearly reports unsupported vision, and it must not fabricate a
different token.

## Goal

Validate real image attachment handling:

- relay extracts and downloads the image resource;
- daemon persists it as an image attachment;
- the next prompt exposes a usable local image path;
- the intern either reads the visible token or truthfully reports inability;
- the product never makes the image silently disappear.

## Why J

Image routing and persistence are F-level, but the user journey is "I sent a
screenshot; can the intern use it?" That requires a live prompt and reply.

## Cost profile

Paid/agent. The case sends a real image attachment and may invoke image-viewing
capability inside the intern turn.

## Key assertions

- Image attachment is available to the intern as a local path.
- Successful path reads the exact visible token.
- Unsupported path is honest and actionable.
- Wrong-token hallucination is a product/model behavior finding.

## Implementation prerequisites

- Real Feishu image attachment driver.
- Tiny image fixture generator.
- Image-use evidence extractor or explicit unsupported classifier.

## Failure classification

- Image route/persistence fails: product bug or Feishu permission gap.
- Local path is missing from prompt context: product bug.
- Agent fabricates image content: product/model behavior finding.
- CI cannot drive image attachment upload: CI helper gap.
