# J_0011 Feishu rich content natural split journey

## Retired reason

Session 12 supervisor review asked to consolidate small feature-point scripts
into broader journey bundles. This outgoing rich-content split coverage is now
folded into `J_0090`, which covers image send, file send, text rollover
boundaries, and oversized delivery failure/fallback in one multi-prompt user
journey.

## Concise proposal

Prompt: ask Codex to first send a short text update, then use the
`feishu-messaging` skill to send a tiny image, then continue with one more text
line.

Look for: the text before the image is finalized, the image is a real Feishu
image message, and the text after the image appears in a newer Feishu message
instead of updating the old pre-image message.

Prompt: repeat with a small file artifact.

Look for: the same split behavior around the file message, with message ids
ordered as text-before, rich content, text-after.

## Goal

Validate the natural user journey for Feishu rich content split:

- the user asks an intern to report progress and send an artifact;
- the intern uses the built-in `feishu-messaging` skill in a real turn;
- image/file sends create rich content boundaries;
- subsequent assistant text rolls over to a new Feishu message;
- report evidence links the real prompt, rich content message id, and follow-up
  text message id.

## Why J

The rich content boundary is not adequately proven by direct hook, daemon, or
Feishu API calls. This case is J-scoped because the intern must naturally use
the `feishu-messaging` skill from a real prompt, then continue speaking after
the rich content send. That is the product behavior the supervisor sees.

## Cost profile

Paid/agent. The case starts a real Codex session, asks it to create small local
artifacts, uses a real skill call, and waits for real Feishu messages.

## Required environment

- Backend: `codex`.
- A case-scoped Feishu group and Codex session.
- Built-in `feishu-messaging` skill available in the Codex farm.
- Real Feishu image/file upload ability through approved policy credentials.
- Message timeline access that can distinguish update-in-place from new
  message ids.

## Key assertions

- Rich content split is triggered by natural agent use of the
  `feishu-messaging` skill.
- Text before rich content is not edited after the image/file is sent.
- Text after rich content uses a newer Feishu message id.
- Both image and file paths are covered.
- Report evidence can reconstruct the message timeline order.

## Implementation prerequisites

- A Feishu timeline helper that records message id, type, create/update time,
  and content kind.
- A skill-output parser for `feishu-messaging` image/file success output.
- A tiny artifact fixture helper that avoids large uploads and secrets.
- A guard that fails if the runner directly calls hook/daemon rich-content
  boundary helpers instead of driving a real agent turn.

## Failure classification

- Text after rich content updates the old pre-rich message: product bug.
- Rich content send succeeds but no boundary is recorded: product bug.
- Skill cannot resolve the case group despite correct `(project, intern)`:
  product bug or policy/registry bug.
- CI sends rich content directly through daemon/API instead of agent skill use:
  CI bug.
- Timeline evidence cannot distinguish update vs new message: CI helper gap.
