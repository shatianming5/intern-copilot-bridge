# J_0097 Helper backend fallback honesty journey

<!-- METADATA:TYPE=J,ID=J_0097,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=9 -->

## Concise proposal

Prompt: start helper with a preferred backend unavailable.

Look for: helper either falls back with an explicit notice or reports the
backend blocker, without pretending the preferred backend is running.

## Goal

Validate helper backend fallback honesty:

- preferred helper backend is unavailable in the case fixture;
- user asks to start helper;
- system uses allowed fallback or reports a precise blocker;
- helper reply names the actual runtime path.

## Why J

Runtime selection contracts are F. This case validates the user-facing helper
conversation after backend selection.

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Prepare helper fixture where preferred backend is unavailable. | Fallback/blocked path is deterministic. |
| 2 | action | Prompt/start helper with the preferred backend. | Helper selection happens. |
| 3 | wait/assert | Wait for helper reply. | Reply says fallback backend used or names blocker. |
| 4 | assert | Check runtime evidence. | Actual backend matches user-visible claim. |
| 5 | assert | Check report. | Report records preferred backend, actual backend, and final answer. |

## Key assertions

- User-visible backend claim matches runtime evidence.
- No fake preferred-backend success.

## Implementation prerequisites

- Helper backend availability fixture.
- Runtime backend evidence collector.

## Failure classification

- Claimed preferred backend while fallback used: product/model behavior bug.
- Silent helper start failure: product bug.
