# J_0089 Bad hook state recovery journey

## Retired reason

Session 12 supervisor review rejected this case as a J journey. Bad hook state
recovery has too many valid implementation strategies, and the agent should not
be asked to determine whether local hook state is correct. Keep any future
coverage at the F/hook invariant level unless the supervisor defines a narrower
user-visible product story.

## Concise proposal

Retired. Do not implement as an active J script.

Historical draft: corrupt a case-scoped hook state, then ask a normal question.

## Goal

Historical idea only; this is not an accepted J goal. The rejected goal was to
validate recovery from bad runtime hook state:

- corrupted hook state is scoped to the case intern;
- next user prompt does not replay a huge old transcript;
- agent either answers normally or reports a precise local-state blocker;
- state repair evidence is recorded.

## Why J

The hook parser itself can be tested in F. This case verifies what a supervisor
sees when a real prompt arrives after bad state.

## Scenario

| # | Type | Operation | Expected |
|---|------|-----------|----------|
| 1 | action | Seed a case-scoped malformed hook state file. | Only the case intern is affected. |
| 2 | action | Prompt a small read-only question with token. | Prompt enters normal delivery path. |
| 3 | wait/assert | Wait for reply or blocker. | No giant old transcript is posted; token is handled. |
| 4 | assert | Check local state. | Bad state is skipped, quarantined, or reported with path. |
| 5 | assert | Check report. | Report includes timeline size, state path, and recovery classification. |

## Key assertions

- No transcript flood.
- Recovery is scoped and diagnosable.

## Implementation prerequisites

- Case-scoped hook state writer.
- Feishu message-size/timeline guard.

## Failure classification

- Old transcript flood in Feishu: product bug.
- Bad state from one intern poisons another intern: product bug.
