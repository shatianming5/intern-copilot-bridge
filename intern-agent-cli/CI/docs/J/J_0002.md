# J_0002 GitHub protected lifecycle journey

<!-- METADATA:TYPE=J,ID=J_0002,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=5 -->

## Concise proposal

Prompt: `分配任务 <task_id>。`

Look for: the task is assigned from the Feishu message, the intern becomes
Working, and no extra task is created.

Prompt: `继续`

Look for: the intern opens a GitHub PR against the protected target branch,
business diff contains only the requested README change, and target revision has
not moved yet.

Prompt: `没问题，可以merge`

Look for: merge happens only after this prompt, target branch advances through
the PR, task closeout completes, intern returns Idle, and no metadata enters the
business PR.

## Goal

Validate the real user lifecycle for a protected GitHub repository using
metadata-branch workspace mode:

- a task exists but is not assigned by precondition actions;
- the user assigns it through a real Feishu prompt;
- the user asks the intern to continue;
- the intern creates a GitHub PR instead of direct-pushing;
- the user explicitly approves merge through Feishu;
- the task closes out and the intern returns Idle.

## Why J

This is J-scoped because assignment, continuation, and merge approval are real
user prompts. F cases can verify provider APIs, registry state, and protected
branch contracts, but they cannot prove that an intern respects user-driven
assignment and approval before merging a GitHub PR.

## Cost profile

Paid/agent. The case starts a real Codex intern session, sends natural prompts,
waits for agent work, and creates a real GitHub PR in a throwaway protected test
repository.

## Required environment

- A lead-prepared GitHub protected test workspace.
- Provider: `github`.
- Workspace mode: `metadata_branch`.
- Backend: `codex`.
- Protected branch rules remain enabled.
- The target branch must be throwaway or otherwise approved for CI merge.
- The case must not add, delete, migrate, or switch workspace mode.

## Case resources

- namespace: `ci_j_0002`
- intern: `intern_ci_j_0002_codex_<run_id>`
- task fixture: `task_ci_j_0002_lifecycle_<run_id>`
- source branch: `ci_j_0002/<task_id>`
- report prefix: `j_0002_github_lifecycle`
- retained scene: the final Feishu group, assignment/continue/approval
  messages, GitHub PR, before/after revisions, diff evidence, task closeout
  evidence, and CI report artifacts remain available for review.

Start-of-case cleanup may remove only `ci_j_0002` resources from previous runs:
case intern, session, task fixture, source branch, open test PR, and report
artifacts. It must not clean lead-owned workspaces or unrelated branches/PRs.

## Setup

The task body should require one small business-only edit, for example:

```text
Please update README.md with one line under "CI protected lifecycle":
J0002 github protected lifecycle verified.

Do not modify .intern_workspace.
```

The runner records:

- baseline task ids;
- target branch `before_rev`;
- workspace provider, repo URL, metadata mode, and target branch;
- unassigned task state;
- intern Idle state before assignment.

## Key assertions

- Assignment source is the Feishu prompt, not precondition or CLI assignment.
- `继续` opens a GitHub PR and does not advance the protected target branch.
- Business PR diff contains the requested README change and no metadata files.
- Merge happens only after the explicit Feishu approval prompt.
- The target branch advances through the PR merge path, not direct push.
- Task closeout completes and intern returns Idle.
- The case never adds, deletes, migrates, or switches the workspace.

## Failure classification

- Task assignment is not tied to the Feishu prompt: product bug or CI fixture bug,
  classified by evidence.
- GitHub workspace is treated as local, Codeup, or non-protected: product bug.
- Target branch moves before approval: product bug.
- Merge happens before approval or without approval: product bug.
- Direct push bypasses PR: product bug.
- Metadata enters the business PR: product bug.
- Cleanup deletes or mutates the lead-prepared workspace: CI bug.
