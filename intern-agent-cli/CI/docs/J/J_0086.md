# J_0086 Goal footer visible state after replace and clear journey

<!-- METADATA:TYPE=J,ID=J_0086,STATUS=ReviewDraft,OWNER=intern_ci_lead,SESSION=10 -->

## Concise proposal

Prompt: set a long-running goal that cannot complete until a case marker
appears, replace it, clear it, then ask a normal follow-up.

Look for: Feishu-visible goal footer/footnote and `/goal status` match the real
current goal, and do not resurrect stale replaced or cleared text.

## Goal

Validate goal footer visibility after lifecycle changes:

- an initial long-running goal becomes visible to the user and remains active
  long enough to inspect;
- Replace confirmation changes the active goal only after approval;
- clear removes the goal from future visible replies;
- later turns do not show stale goal state.

## Why J

The goal API and TUI markers are F-level. This case verifies the supervisor's
actual Feishu experience while the agent continues normal conversation.

## Key assertions

- Cancel/Replace state is reflected in user-visible footer/status.
- The initial and replacement goals are long-running enough to inspect before
  completion; immediate completion is a case design failure.
- Feishu footnote/footer text matches `/goal status` after set, replace, and
  clear.
- Cleared goals do not reappear on later turns.

## Implementation prerequisites

- Goal status/footer timeline extractor.
- Replace confirmation card driver.
- Long-running marker fixture that keeps the goal active until the case says the
  marker exists.

## Failure classification

- Stale goal footer after replace/clear: product bug.
- Goal changed before confirmation: product bug.
- Goal completes before the status/footer can be inspected because the fixture
  was not long-running: CI script bug.
